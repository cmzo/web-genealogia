# Data Model: Data Pipeline & Foundation

**Feature**: 001 — Data Pipeline & Foundation
**Date**: 2026-04-27

This document defines the exact JSON schemas for the three generated artefacts written
to `data/` by the CI pipeline, and the GEDCOM output structure.

---

## 1. `data/individuals.json`

Array of Individual objects. Source: Google Sheet 1.

```json
[
  {
    "id":          "string  — permanent unique identifier, used as URL slug",
    "first_name":  "string",
    "last_name":   "string",
    "birth_date":  "string (YYYY-MM-DD) | null",
    "birth_place": "string | null",
    "death_date":  "string (YYYY-MM-DD) | null",
    "death_place": "string | null",
    "father_id":   "string (references id) | null",
    "mother_id":   "string (references id) | null",
    "notes":       "string | null"
  }
]
```

**Constraints**:
- `id` MUST be non-empty and unique across all records.
- `birth_date` and `death_date` MUST conform to `YYYY-MM-DD` if present, or be `null`.
- `father_id` and `mother_id` MUST reference a valid `id` in the same array, or be `null`.
  A dangling reference (referenced ID not present) MUST NOT cause a pipeline failure;
  it is silently retained as-is.
- All other string fields may be empty string `""` or `null`; both are treated as absent
  by the frontend.
- Key order within each object MUST follow the schema above for readability.

**Example record**:
```json
{
  "id": "fcl-001",
  "first_name": "Francisco",
  "last_name": "Clemenzo",
  "birth_date": "1872-03-15",
  "birth_place": "Ardón, León, España",
  "death_date": "1951-11-02",
  "death_place": "Buenos Aires, Argentina",
  "father_id": "fcl-000",
  "mother_id": null,
  "notes": "Emigró a Argentina en 1903 junto a su hermano Victorino."
}
```

## 2. `data/files.json`

Array of MediaFile objects. Source: Google Sheet 2.

```json
[
  {
    "file_id":       "string — permanent unique identifier",
    "individual_id": "string — references individuals.json[].id",
    "file_type":     "\"image\" | \"document\"",
    "url":           "string — publicly accessible URL",
    "caption":       "string",
    "date":          "string (YYYY-MM-DD) | null"
  }
]
```

**Constraints**:
- `file_id` MUST be non-empty and unique.
- `individual_id` MUST reference a valid `id` in `individuals.json`, or the pipeline
  logs a warning but does NOT drop the record.
- `file_type` MUST be exactly `"image"` or `"document"` (lowercase); values from the
  Sheet are normalised to lowercase by the pipeline script.
- `url` MUST be a non-empty string; the pipeline does not validate reachability.
- `caption` MUST be a non-empty string (used as `alt` text per the constitution).
- `date` follows `YYYY-MM-DD` or `null`.

**Example record**:
```json
{
  "file_id": "img-0042",
  "individual_id": "fcl-001",
  "file_type": "image",
  "url": "https://raw.githubusercontent.com/cmzo/genealogia/main/assets/images/posts/francisco-1903.webp",
  "caption": "Francisco Clemenzo, fotografía de despedida, Ardón 1903",
  "date": "1903-01-01"
}
```

## 3. `data/search-index.json`

Reduced array for client-side Fuse.js search. Generated by `scripts/generate-search-index.js`
from `individuals.json`.

```json
[
  {
    "id":          "string — same as individuals.json[].id",
    "first_name":  "string",
    "last_name":   "string",
    "birth_place": "string | null",
    "birth_date":  "string (YYYY-MM-DD) | null"
  }
]
```

**Constraints**:
- Every record in `individuals.json` MUST have a corresponding entry in `search-index.json`.
- Fields are a strict subset of the Individual schema; no additional fields are included.
- The array order MUST match `individuals.json` (same sort, derived by iteration).

**Example record**:
```json
{
  "id": "fcl-001",
  "first_name": "Francisco",
  "last_name": "Clemenzo",
  "birth_place": "Ardón, León, España",
  "birth_date": "1872-03-15"
}
```

**Fuse.js configuration** (reference for Feature 005/006 implementation):
```js
const FUSE_OPTIONS = {
  keys: ['first_name', 'last_name', 'birth_place'],
  threshold: 0.35,   // fuzzy tolerance: ~1 character substitution
  includeScore: true,
  minMatchCharLength: 2,
};
```

## 4. `public/export/family.ged` — GEDCOM 5.5.1

Generated by `scripts/generate-gedcom.js` from `individuals.json`. Format is
plain text; no binary encoding. Line endings: CRLF (GEDCOM standard).

**Record types produced**:

| Tag | Source |
|-----|--------|
| `HEAD` | Fixed header (charset UTF-8, GEDCOM 5.5.1, source "clemenzo-de-ardon") |
| `INDI` | One record per individual |
| `FAM` | One record per unique father-mother pair that has ≥1 child |
| `TRLR` | Fixed trailer |

**INDI record structure**:
```
0 @I{id}@ INDI
1 NAME {first_name} /{last_name}/
2 GIVN {first_name}
2 SURN {last_name}
1 BIRT
2 DATE {birth_date}      ← omit line if null
2 PLAC {birth_place}     ← omit line if null
1 DEAT Y                  ← only if death_date or death_place present
2 DATE {death_date}      ← omit line if null
2 PLAC {death_place}     ← omit line if null
1 NOTE {notes}            ← omit if null; long notes continue on CONT lines
```

**FAM record structure** (one per unique `{father_id, mother_id}` combination):
```
0 @F{n}@ FAM
1 HUSB @I{father_id}@   ← omit if father_id null
1 WIFE @I{mother_id}@   ← omit if mother_id null
1 CHIL @I{child_id}@    ← one line per child
```

**INDI ↔ FAM cross-reference**:
Each `INDI` record that belongs to a FAM receives:
```
1 FAMS @F{n}@   ← for each FAM where this individual is HUSB or WIFE
1 FAMC @F{n}@   ← for the FAM where this individual is a CHIL
```

**Encoding**: UTF-8 throughout. The HEAD record declares `1 CHAR UTF-8`.

## 5. Relationships & Derivations

```
Google Sheet 1
    │
    ▼
scripts/fetch-sheets.js
    │
    ├──► data/individuals.json
    │         │
    │         ├──► scripts/generate-search-index.js ──► data/search-index.json
    │         │
    │         └──► scripts/generate-gedcom.js ──► public/export/family.ged
    │
Google Sheet 2
    │
    ▼
scripts/fetch-sheets.js ──► data/files.json
```

All four output files are generated in a single workflow run. If `individuals.json`
generation fails, the pipeline MUST abort rather than write partial data.
