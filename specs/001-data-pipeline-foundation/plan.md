# Implementation Plan: Data Pipeline & Foundation

**Branch**: `001-data-pipeline-foundation` | **Date**: 2026-04-27
**Spec**: [spec.md](spec.md)
**Input**: Feature specification from `specs/001-data-pipeline-foundation/spec.md`

## Summary

Build the invisible infrastructure that makes all other site features possible:
a GitHub Actions pipeline that fetches data from two Google Sheets, writes three
static JSON artefacts (`individuals.json`, `files.json`, `search-index.json`) and
a GEDCOM export file, commits them to `main`, and deploys to GitHub Pages — all
without any runtime API calls from the visitor's browser. A Service Worker provides
offline support after the first visit.

## Technical Context

**Language/Version**: Node.js ≥ 18 (pipeline scripts only; zero JS in browser bundle
for this feature)
**Primary Dependencies**: `@googleapis/sheets` (pipeline only, never bundled for browser)
**Storage**: Static JSON files committed to the repository under `data/`
**Testing**: Manual smoke tests per `quickstart.md`; no automated test suite for v1
**Target Platform**: GitHub Actions (Ubuntu runner) + GitHub Pages (static hosting)
**Project Type**: CI/CD data pipeline + static site asset generation
**Performance Goals**: Pipeline completes in under 3 minutes; JSON files load in <500ms
on broadband (file size driven by data volume, typically <200KB each)
**Constraints**: No client-side API keys; no runtime Sheets calls; `[skip ci]` on
auto-commits to prevent infinite loop; Service Worker cache name `clemenzo-v1`
**Scale/Scope**: Hundreds of individuals; tens to low-hundreds of media files

## Phase 1 — Pipeline Implementation

**Prerequisites**: research.md complete ✅

### 1.1 `scripts/package.json`

Minimal Node.js project for pipeline scripts only. Single dependency:
`@googleapis/sheets`. No devDependencies needed for v1.

```json
{
  "name": "clemenzo-pipeline",
  "version": "1.0.0",
  "type": "module",
  "dependencies": {
    "@googleapis/sheets": "^8"
  }
}
```

### 1.2 `scripts/fetch-sheets.js`

Responsibilities:
1. Authenticate with Google Sheets API v4 using `GOOGLE_SERVICE_ACCOUNT_JSON`
   (parsed from env var, never from a file in the repo).
2. Read Sheet 1 (individuals) — first row is the header; map remaining rows to
   Individual objects using the schema in `data-model.md`.
3. Read Sheet 2 (files) — same header-row pattern; map to MediaFile objects.
4. Normalise: trim whitespace; convert empty strings to `null`; lowercase
   `file_type`; validate date format (log warning, do not fail, on bad dates).
5. **Safety check**: if either sheet returns 0 rows (excluding header), abort
   with exit code 1 — do NOT overwrite existing JSON with an empty array.
6. Write `data/individuals.json` and `data/files.json` (pretty-printed, 2-space
   indent for readable git diffs).
7. Call `generate-search-index.js` and `generate-gedcom.js` in sequence.

Environment variables consumed:
- `GOOGLE_SERVICE_ACCOUNT_JSON` — full JSON key (raw JSON string)
- `SHEET1_ID` — Google Sheet 1 spreadsheet ID
- `SHEET2_ID` — Google Sheet 2 spreadsheet ID

### 1.3 `scripts/generate-search-index.js`

Reads `data/individuals.json` (already written by fetch-sheets.js). Produces
`data/search-index.json` by mapping each Individual to the 5-field reduced schema
(`id`, `first_name`, `last_name`, `birth_place`, `birth_date`). Pretty-printed.

### 1.4 `scripts/generate-gedcom.js`

Reads `data/individuals.json`. Produces `public/export/family.ged`.

Algorithm:
1. Assign each individual a GEDCOM pointer: `@I{id}@`.
2. Build a FAM map: group individuals by unique `{father_id, mother_id}` pairs
   (skip pairs where both are null). Assign each FAM a sequential pointer `@F1@`, `@F2@`, etc.
3. For each individual, write an `INDI` record (see `data-model.md` for exact format).
   Include `FAMS` and `FAMC` cross-references.
4. For each FAM, write a `FAM` record.
5. Write header and trailer.
6. Use CRLF line endings; UTF-8 encoding.
7. Write to `public/export/family.ged`. Create `public/export/` if it does not exist.

### 1.5 `.github/workflows/refresh-data.yml`

```yaml
name: refresh-data

on:
  push:
    branches: [main]
  schedule:
    - cron: '0 3 * * *'

jobs:
  refresh:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          token: ${{ secrets.GITHUB_TOKEN }}

      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'
          cache-dependency-path: scripts/package-lock.json

      - name: Install pipeline deps
        run: npm ci
        working-directory: scripts

      - name: Fetch Sheets data
        env:
          GOOGLE_SERVICE_ACCOUNT_JSON: ${{ secrets.GOOGLE_SERVICE_ACCOUNT_JSON }}
          SHEET1_ID: ${{ secrets.SHEET1_ID }}
          SHEET2_ID: ${{ secrets.SHEET2_ID }}
        run: node scripts/fetch-sheets.js

      - name: Commit generated artefacts
        run: |
          git config user.name  "github-actions[bot]"
          git config user.email "github-actions[bot]@users.noreply.github.com"
          git add data/ public/export/
          git diff --cached --quiet || \
            git commit -m "data: refresh from Google Sheets [skip ci]"
          git push
```

The workflow only commits and pushes when there are actual changes
(`git diff --cached --quiet` guard). GitHub Pages deployment is triggered
automatically by the push to `main`.

## Phase 2 — Service Worker & Offline Validation

**Prerequisites**: Phase 1 complete; `data/*.json` exist in the repo.

### 2.1 `sw.js` (site root)

Cache name: `clemenzo-v1`.

Strategy:
- **Cache-first**: `data/*.json`, all CSS/JS/image/font assets
- **Network-first**: HTML pages (`.html`, navigations to `/`)
- **Default**: Network with cache fallback

On `install`: pre-cache the three data JSON files.
On `activate`: delete all caches with names other than `clemenzo-v1`.

See Phase 1 design for the full SW implementation reference.

### 2.2 SW Registration

Registered from `assets/js/base.js` with feature detection:

```js
if ('serviceWorker' in navigator) {
  window.addEventListener('load', () => {
    navigator.serviceWorker.register('/sw.js')
      .catch(err => console.warn('SW registration failed:', err));
  });
}
```

`sw.js` MUST be at the site root so its scope covers the full origin.

covering local testing, CI run verification, and offline/SW smoke tests.
