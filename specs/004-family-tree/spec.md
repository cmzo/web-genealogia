# Feature Specification: Interactive Family Tree

**Feature Branch**: `004-family-tree`
**Created**: 2026-04-27
**Status**: Draft
**Input**: User description — Feature 004

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Desktop Visitor Explores the Tree (Priority: P1)

A family member opens the family tree page on a desktop browser. They can see
a portion of the tree, pan in any direction to reveal more relatives, zoom in
to read names and dates clearly, and zoom out to get an overview of the whole
family.

**Why this priority**: The interactive tree is the centrepiece of the site.
If it does not work well on desktop, the primary value proposition fails.

**Independent Test**: Open the tree page with 100+ individuals; a visitor can
pan, zoom in, zoom out, and locate a specific person by name within 30 seconds.

**Acceptance Scenarios**:

1. **Given** a visitor opens the tree page on a 1280px desktop, **When** the
   page loads, **Then** a visual representation of the family tree is visible
   and the visitor can immediately begin navigating it by clicking and dragging.
2. **Given** the visitor is exploring the tree, **When** they use a scroll
   wheel or pinch gesture, **Then** the tree zooms in and out smoothly.
3. **Given** the tree is zoomed in, **When** the visitor pans, **Then** the
   rest of the tree becomes visible and relationships (connecting lines)
   remain clear.

---

### User Story 2 — Navigate to an Individual's Profile (Priority: P1)

A visitor spots a person in the tree they want to learn more about. They click
that person's node and are taken directly to the individual's profile page.

**Why this priority**: The tree is a navigation surface, not just a display.
Clicking through to profiles is its primary interactive purpose.

**Independent Test**: Click on 10 different nodes; each one navigates to the
correct individual profile with no errors.

**Acceptance Scenarios**:

1. **Given** a visitor clicks on a person's node in the tree, **When** the
   click registers, **Then** the visitor is navigated to that person's profile
   page at `/profiles/[id]`.
2. **Given** the visitor navigates to a profile and presses the browser Back
   button, **When** they return, **Then** the tree is in the same state it was
   before they clicked (zoom level and position preserved).

---

### User Story 3 — Tree Handles Missing Parental Data Gracefully (Priority: P1)

Many individuals in the tree have one or both parents unknown or unrecorded.
The tree renders without errors, gaps in the display look intentional, and the
rest of the tree is unaffected.

**Why this priority**: Missing data is the normal state for genealogical records,
not an exception. A tree that breaks on missing data is unusable.

**Independent Test**: Load the tree with at least 10 individuals who have no
recorded father or mother; the tree renders without errors or blank spaces that
disrupt layout.

**Acceptance Scenarios**:

1. **Given** an individual has no recorded father, **When** the tree renders,
   **Then** that individual's node appears with an open-ended branch (or no
   branch) on the paternal side — no error message, no broken layout.
2. **Given** an individual has no parents at all, **When** the tree renders,
   **Then** that individual appears as a root node without any parent connectors.

---

### User Story 4 — Mobile Visitor Navigates the Family (Priority: P2)

A visitor on a phone opens the tree page. Rather than a small, unreadable
version of the desktop tree, they see a simplified view that lets them browse
the family — for example, by starting from one person and stepping up or down
through their lineage.

**Why this priority**: The full interactive tree is unusable on a small screen.
Mobile visitors still need a way to explore relationships.

**Independent Test**: Open the tree page on a 375px screen; a visitor can reach
any individual's profile by navigating through the simplified view.

**Acceptance Scenarios**:

1. **Given** a visitor opens the tree page on a phone, **When** the page loads,
   **Then** a simplified ancestor/descendant navigation view is shown (not the
   full zoomable tree).
2. **Given** the mobile visitor is viewing one individual, **When** they tap
   on a parent or child, **Then** the view updates to show that relative's
   lineage.
3. **Given** the mobile visitor finds the person they were looking for, **When**
   they tap their name, **Then** they are taken to that individual's profile.

---

### User Story 5 — Download the Genealogy Export (Priority: P3)

A family member wants to import the family data into Ancestry or FamilySearch.
They find the download link on the tree page and download the GEDCOM file.

**Why this priority**: The export file is generated by Feature 001; this feature
only needs to surface a download link. Low effort, but explicitly required.

**Independent Test**: The GEDCOM download link is visible on the tree page on
desktop without scrolling; clicking it downloads a valid `.ged` file.

**Acceptance Scenarios**:

1. **Given** a visitor is on the tree page, **When** they look for the export
   option, **Then** a download link for the GEDCOM file is visible without
   scrolling on a 1280px screen.
2. **Given** a visitor clicks the download link, **When** the download starts,
   **Then** they receive the same `.ged` file generated by the data pipeline.

---

### Edge Cases

- What if the tree contains a cycle (person A is ancestor of person B and
  also a descendant)? The tree MUST detect and break cycles gracefully rather
  than crashing or rendering infinitely.
- What if the tree contains only one person? The tree MUST render that single
  node without errors.
- What if two individuals share the same name? Each node is identified by `id`,
  not by name — duplicate names do not cause confusion.
- What happens at extreme zoom levels (very zoomed in or very zoomed out)? The
  tree MUST clamp zoom to a sensible range so nodes never become invisible or
  invisible due to overflow.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: The tree MUST be generated automatically from `data/individuals.json`
  (produced by Feature 001) — the author does not draw or configure it manually.
- **FR-002**: The desktop tree MUST support pan (click-drag) and zoom
  (scroll wheel or pinch) navigation.
- **FR-003**: Each individual node MUST display the person's full name and, where
  available, birth and death years.
- **FR-004**: Parent-child relationships MUST be represented by visible connecting
  lines between nodes.
- **FR-005**: Clicking or tapping an individual node MUST navigate to that
  person's profile page.
- **FR-006**: Missing parent data MUST be handled gracefully — open-ended
  branches, no errors, no broken layout.
- **FR-007**: The tree MUST remain interactive and smooth with 500 or more
  individuals rendered.
- **FR-008**: On screens narrower than 768px, the full interactive tree MUST be
  replaced with a simplified ancestor/descendant navigation view.
- **FR-009**: The simplified mobile view MUST allow navigating to any individual
  and linking to their profile.
- **FR-010**: The page MUST include a visible download link for the GEDCOM export
  file (produced by Feature 001).
- **FR-011**: Tree nodes MUST be keyboard-navigable (focusable, activatable via
  Enter/Space).
- **FR-012**: Zoom MUST be clamped to a sensible range; nodes MUST never be
  fully invisible due to zoom extremes.

### Key Entities

- **Tree node**: Visual representation of an Individual. Displays: full name,
  birth year (if known), death year (if known). Links to: `/profiles/[id]`.
- **Relationship edge**: A visual connector between a parent node and a child
  node. Derived from `father_id` and `mother_id` fields.
- **Viewport**: The visible area of the tree canvas. Can be panned and zoomed
  by the visitor.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: A visitor can locate any named individual in the tree within
  30 seconds on a desktop with 100+ people rendered.
- **SC-002**: The tree remains visually smooth and responsive to pan/zoom
  interactions with 500 or more individuals (no input lag noticeable to the
  visitor).
- **SC-003**: 100% of individual node clicks navigate to the correct profile
  page.
- **SC-004**: The tree renders without any error state or broken layout when
  at least 20% of individuals have one or both parents missing.
- **SC-005**: A mobile visitor can navigate to any individual and reach their
  profile within 5 taps or fewer using the simplified view.
- **SC-006**: The GEDCOM download link is visible on the tree page without
  scrolling on a 1280px desktop screen.

## Assumptions

- `data/individuals.json` is produced and kept current by Feature 001; this
  feature consumes it as a read-only data source.
- The genealogy export file (`.ged`) is also produced by Feature 001 and stored
  at a predictable static path; this feature only links to it.
- "Smooth" interaction with 500+ people means no perceptible input lag on a
  mid-range desktop; virtualisation of off-screen nodes is the expected
  implementation approach (per the constitution's performance rules).
- The tree visualisation layout is hierarchical (top-down or left-right
  generational layout); the exact layout algorithm is a planning/implementation
  decision.
- Sibling relationships are visually implied by shared parent nodes; explicit
  sibling connectors are out of scope for v1.
- The browser Back button behaviour (restoring tree state) is best-effort;
  exact scroll/zoom restoration is a nice-to-have, not a hard requirement.
